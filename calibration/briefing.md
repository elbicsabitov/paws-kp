# Paws Calibration — задача для аннотатора

**Время:** ~2.5 часа (можно одним блоком или 2×1.5ч)
**Где:** один HTML файл, открываешь в браузере
**Сколько пар:** 30 (одинаковые у тебя, Sofia, Diana, Milana — это **multi-annotator calibration**)

---

## Что нужно сделать

Прочитать 30 пар «сообщение клиента → ответ менеджера» из реальных переписок Paws.kz. По каждой паре поставить оценку **PASS / FAIL / N/A** в 5 колонках качества + уровень уверенности 1-3 + комментарий (обязателен на FAIL).

В конце нажать «Скачать результаты» → отправить файл .json Эльбику в Telegram.

---

## Зачем это нужно (важно понимать)

Бот учится сам через автоматическую оптимизацию. Чтобы измерить «качество» — нужен LLM-судья. Чтобы судья судил правильно, он должен научиться у **группы** менеджеров, не у одного.

**Поэтому 3 annotator'а** (Sofia, Diana, Milana) labelят одни и те же 30 пар независимо. Где все 3 совпали — ground truth. Где разошлись — обсуждаем 5 минут все вместе, переформулируем критерий.

Без вашей разметки все «+5% качества бота» = vibes. Поэтому 2.5 часа от каждой = критично.

---

## Почему 3 annotator'а вместо одного

**Single-annotator bias** — главная проблема eval. Если только Sofia label'ит — мы оптимизируем под её **идеал**, не под manager-reality. Бот будет лучше по её rubric, но хуже по conversion в Bitrix.

3 senior manager'а с разным углом зрения = взаимная коррекция. Где совпали 3/3 — это объективная правда. Где 2/1 — пограничный случай, обсуждаем.

---

## 5 критериев (что оценивать в ответе менеджера)

> 💡 **Главное правило:** оцениваем **«как менеджеры реально работают»**, не «как должно быть в идеале». Если manager Иван/Диана/Милана реально пишет 5-словный ответ без долгого объяснения pricing_canonical в каждом турне — это **PASS**. Идеал ≠ реальность.

### 1. Анкета двигается?

Менеджер двигает диалог к закрытию или топчется?
- ✅ **PASS** — клиент новый → manager задал имя питомца ИЛИ city ИЛИ даты (любой следующий шаг)
- ❌ **FAIL** — manager третий раз спрашивает «какой питомец?» хотя клиент уже сказал
- ⚪ **N/A** — анкета уже полностью заполнена в предыдущих turns

### 2. Тон менеджера

Тёплый, по-русски, как реальный менеджер? НЕ корпоративный, НЕ робот?
- ✅ **PASS** — «Какая милая собачка! Конечно поможем. Как зовут вашего пушистика?»
- ❌ **FAIL** — «Согласно нашим стандартам обслуживания…» (корп) ИЛИ «Я ИИ-ассистент» (robot)
- ⚪ **N/A** — реакция на 1-словный ответ типа «ок» — тон сложно оценить

### 3. Возражение отработано?

Если клиент возразил («дорого / подумаю / сравниваю») — manager отработал возражение или просто повторил цену?
- ✅ **PASS** — «дорого» → «Понимаю. У нас вакцинация-чек + страховка ст.329. Альтернатива за 6500?»
- ❌ **FAIL** — «дорого» → «Цена 9500» (просто повторил)
- ⚪ **N/A** — возражения в этом turn не было

### 4. Попытка закрыть готового

Клиент явно готов («давайте оформим / беру») — manager двинул к оплате?
- ✅ **PASS** — «давайте оформим» → «Скиньте паспорт питомца + адрес, отправлю Каспи»
- ❌ **FAIL** — «беру» → «А когда удобно?» (упустил готового)
- ⚪ **N/A** — клиент не выразил готовность платить

### 5. Цель этого turn достигнута?

Что-то полезное произошло? (deal closed / info collected / objection resolved / next step initiated)
- ✅ **PASS** — клиент спросил цену → manager назвал цену + предложил следующий шаг
- ❌ **FAIL** — клиент задал вопрос → manager пожелал хорошего дня (ничего не достиг)
- ⚪ **N/A** — turn чисто organisational («сейчас гляну»)

## Только эти 5 критериев (другие 5 deterministic)

Раньше я говорил о 10 критериях. **Теперь только 5** — остальные 5 (точность цены / надбавки / phase / safety / факты) проверяет бот автоматически через regex + lookup. **Annotator вы — только на 5 субъективных.**

Это снижает время с 8h до 2.5h без потери качества.

## Workflow в web app (что увидишь когда откроешь HTML)

1. **Welcome screen** — приветствие, 30 пар, ~2.5ч, кнопка «Начать»
2. **Pair labeling** (повторяется 30 раз):
   - Слева: сообщение клиента
   - Справа: ответ менеджера
   - Снизу: 5 критериев × 3 кнопки (PASS/FAIL/N/A) каждый
   - Confidence: 1 (уверена) / 2 (сомневаюсь) / 3 (критерий неясен)
   - Comment (обязателен только на FAIL): 1-2 строки почему
   - «Дальше →» (заблокирована минимум 20 секунд + пока не ответишь все 5 + confidence + comment if FAIL)
3. **Section break** каждые 6 пар — небольшой перерыв предлагается (опционально)
4. **Финал** — статистика + кнопка «Скачать результаты»

**Прогресс автосохраняется в браузере.** Можно паузу + закрыть вкладку + потом снова открыть HTML файл — продолжишь с того же места.

**Подсказка:** в каждом критерии есть кнопка «примеры» — открывает PASS/FAIL/N/A примеры в попапе.

## Если сомневаешься — что делать

| Ситуация | Что ставить |
|---|---|
| Критерий не применим к этой паре | **N/A** |
| FAIL «немного» (borderline) | **FAIL** + comment «borderline, пограничный» + confidence 2 |
| Не понимаешь критерий | **N/A** + confidence 3 → потом я переформулирую |
| 3+ подряд «критерий неясен» (confidence=3) | **СТОП** → Telegram Эльбику → исправлю на лету |
| Сомневаешься PASS vs FAIL | смотри строже — лучше FAIL+comment чем PASS из лояльности |
| client_msg обрезан / нечитаемый | **N/A для всех 5** + comment «client_msg unreadable» |

**Главное:** твоё честное «не знаю / N/A» лучше угаданного PASS. Я могу работать с N/A; не могу — с misclassified PASS.

## Если думаешь «менеджеры так не пишут»

Это самая ценная reaction! Поставь **FAIL** + в comment напиши конкретно: **«manager Иван так не пишет, обычно говорит «X»»**. Чем конкретнее, тем лучше.

Эти комментарии пойдут в обсуждение между всеми 3-мя annotator'ами после labeling. Если двое из вас (Diana + Milana, Diana + Sofia, Milana + Sofia) согласны — переформулируем критерий или wiki.

**Не пиши «как должно быть» в comment** — пиши «как реальные менеджеры **реально** пишут».

## После того как сдала JSON

1. Я получаю 3 файла (твой + Diana + Milana)
2. Считаю **Fleiss κ** (3-way inter-annotator agreement) для каждого critерия
3. Где κ ≥ 0.7 → criterion well-formed, используем как ground truth
4. Где κ < 0.7 → criterion ambiguous, переписываю формулировку, прошу всех 3-х lookat 5 spec примеров (5-10 мин)
5. Когда все 5 critериев κ ≥ 0.7 → calibration COMPLETE, я начинаю обучать LLM-судью

**После этого участие annotator'ов:**
- 30 минут: Excel sanity check после Иван clarification
- 5 минут: disputed services decision (Постоянный присмотр / Дневная няня / Няня на сутки)
- 2-4ч/неделю **только для Sofia**: sandbox spot-check (drift detection)

Diana и Milana дальше уже не нужны (их работа — calibration only).

**Total caching commitment:** Sofia ~17-25h за 4 недели, Diana и Milana по ~3h одноразово.

## Технические детали для open

1. Скачай файл `index_{твоё_имя}.html` (Sofia / Diana / Milana — каждая своя версия с её именем в заголовке, но **30 пар идентичны у всех 3-х**)
2. Дабл-клик → откроется в браузере (Chrome / Safari / Firefox любой)
3. Работает offline — интернет нужен только в самом начале (Tailwind CSS из CDN)
4. Прогресс сохраняется в этом браузере (localStorage). Не открывай в incognito mode.
5. По завершении — кнопка «Скачать» → файл `paws_calibration_Sofia_2026-04-28.json` появится в Downloads
6. Отправь .json Эльбику в Telegram

**Если случайно закрыла без сохранения** — данные в браузере останутся. Открой тот же HTML файл снова — продолжит откуда остановилась.
